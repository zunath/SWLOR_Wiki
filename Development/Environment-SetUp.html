Target audience

This guide is targeted towards developers and contributors who want to create areas, creatures, items, write code, add models, and so on. If you want to contribute to the development of the server, you're in the right place!

 

Prerequisites

  • Git GUI. You're welcome to use another client to perform the steps below but I will assume you're doing it from command line.
  • Neverwinter Night: Enhanced Edition (earlier versions, such as 1.69 will NOT work)
  • The “SWLOR In a Box” package. For the purposes of this guide, we will assume you have already extracted this to your NWN directory.
  • If you will be contributing code, Visual Studio Community (You do NOT want Visual Studio Code. It should work, but we do not officially support it.)

 

Development environment set-up

This guide will walk you through setting up your development environment.

  1. Make a GitHub account if you do not already have one.
  2. Go to the SWLOR_NWN repository here: https://github.com/zunath/SWLOR_NWN
  3. Click on the Fork button. This will create a copy of the code base tied to your GitHub account. You will be able to create branches, submit pull requests, and more. All of this will be required to submit your changes to the master SWLOR code base.
  4. Go to the SWLOR_Haks repository here: https://github.com/zunath/SWLOR_Haks
  5. Click on the Fork button. This will create a copy of the hakpak content (models, textures, sounds, etc.) tied to your GitHub account.
  6. Create a folder on your PC to store your projects. My recommendation is C:/Projects/ but anywhere will work, really. Adjust the following commands as necessary if you pick a different location.
  7. Open command prompt
  8. At command line, enter cd C:/Projects/ to change your directory to your projects folder.
  9. At command line, enter git clone <SWLOR_NWN url> replacing <SWLOR_NWN url> with the URL to your SWLOR_NWN project on GitHub. For example: git clone https://github.com/myusername/SWLOR_NWN.git
  10. At command line, enter git clone <SWLOR_Haks url> replacing <SWLOR_Haks url> with the URL to your SWLOR_Haks project on GitHub. For example: git clone https://github.com/myusername/SWLOR_Haks.git
  11. Both of these projects may take a while to clone due to their size. Please be patient and let it run to completion.
  12. At command line, enter cd C:/Projects/SWLOR_NWN/ to change your directory to the code base repository.
  13. At command line, enter git checkout master to change your branch to the master branch.
  14. At command line, enter cd C:/Projects/SWLOR_Haks/ to change your directory to the hakpak content repository.
  15. At command line, enter git checkout master to change your branch to the master branch.

At this point you should have your Git repositories set up and are able to start working on features.

 

Keeping your fork in sync

You will want to keep your fork of both repositories in sync with the master copies to avoid merge conflicts. To do so, please refer to this guide.

 

Creating a feature branch

Every feature you work on should have its own feature branch. This ensures that you can organize your changes and eventually submit them as part of a Pull Request. Follow the steps below to do this.

  1. Ensure your fork is in sync and your local copy is updated. Refer to Git documentation on Google if you need help on this.
  2. Open command prompt
  3. At command line, enter cd C:/Projects/SWLOR_NWN/ to change your directory to the code base repository.
  4. At command line, enter git branch feature/<feature name> replacing <feature name> with a dash-delimited name of the feature. For example: git branch feature/my-awesome-new-feature
  5. At command line, enter cd C:/Projects/SWLOR_Haks/ to change your directory to the hakpak content repository.
  6. At command line, enter git branch feature/<feature name> replacing <feature name> with the same name you used in step 3.
  7. At this point you have a feature branch in both the code repository and the hakpak content repository. You should be ready to make changes.

 

Committing and pushing your changes

When you are finished making changes or simply want to save your progress use the following steps to commit and then push them to GitHub.

To be clear: When you commit changes, they are only being stored on your local repository. They will not show up until you push them to the remote repository located on GitHub. Follow the steps below to do this.

Also note: If you've only made changes in one repository, it is not necessary to commit and push in both. Only commit/push in the repositories which have changed files. We are assuming you have changes in both. Skip steps as necessary.

  1. Open command prompt
  2. At command line, enter cd C:/Projects/SWLOR_NWN/ to change your directory to the code base repository.
  3. At command line, enter git commit and, when prompted, enter a message explaining your changes. Realize that this information can be read by all contributors so it should be as clear and descriptive as possible. Focus on the WHY for the change, rather than the WHAT.
  4. At command line, enter git push to push the changes to your repository on GitHub.
  5. At command line, enter cd C:/Projects/SWLOR_Haks/ to change your directory to the hakpak content repository.
  6. Repeat steps 3 and 4.

 

Submitting a pull request

At this point you should have changes which are committed and pushed to your repository in GitHub. We now need to get your changes merged into the master copy. This is performed by doing a Pull Request. Follow the steps below to do this.

  1. Log into GitHub.
  2. Go to your code repository. This will more than likely be located at a URL such as: https://github.com/myusername/SWLOR_NWN.git
  3. Click on the Pull Requests button.
  4. Click on the New Pull Request button.
  5. Ensure your source is the feature branch you previously pushed up. For example, feature/my-awesome-new-feature
  6. Enter a description for the pull request. Please be as detailed as possible.
  7. A maintainer will review your pull request. They will ask you to change things via comments if needed.
  8. When the code review is completed, your pull request will be merged into the master repository. You're free to start over with a new feature.
